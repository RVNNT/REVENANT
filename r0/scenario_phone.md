# COLDSTART — Сценарій PHONE (Android + iPhone)
## Протокол відновлення цивілізації з нуля, стартова точка: смартфон

---

## ЗАГАЛЬНА ІДЕЯ
Смартфон (Android або iPhone) — перша нода відновлення цивілізації.
Точка входу в повний цикл відновлення:
живлення → Linux середовище → комунікація клану → регіональна мережа → інфраструктура → сервіси → виробництво.

**Android** — основна платформа. Через Termux дає повноцінне Linux середовище, більше свободи для mesh мереж і низькорівневих інструментів. Підтримує USB OTG для підключення зовнішніх модулів (LoRa, програматори STM32/ESP32).

**iPhone** — повноправний учасник на всіх фазах, окрім задач що потребують USB (підключення LoRa модулів, програматорів STM32/ESP32, зовнішнього заліза). Все інше — компіляція, комунікація, сервіси — працює так само як на Android (з джейлбрейком — повний стек, без — обмежено але придатно).

**Правило USB:** якщо крок потребує підключення зовнішнього пристрою через USB → тільки Android. Все інше → Android + iPhone.

## ТЕХНІЧНІ СТЕКИ

### Android — з KVM (unlocked bootloader + кастомне ядро)
Найпотужніший варіант. Апаратна віртуалізація через KVM.

- **Хост:** Termux + Termux:Boot + Termux:Widget
- **VM:** QEMU + KVM → Debian ARM64
- **Веб-розробка:** Node.js, npm, pm2, nginx, Docker, git
- **Системна розробка:**
  - gcc, g++, clang, make, cmake, ninja
  - Linux kernel source + build tools
  - GDB, Valgrind, strace, ltrace
  - Wireshark/tshark, tcpdump
  - STM32: arm-none-eabi-gcc, OpenOCD, STM32CubeMX (через Wine або нативний CLI)
  - ESP32: ESP-IDF toolchain, esptool.py
  - QEMU для емуляції ARM/RISC-V без реального заліза
  - Buildroot або Yocto для збірки кастомних Linux дистрибутивів
- **IDE:** code-server (localhost:8080) або Zed/Sublime через X11/VNC або Neovim
- **Мережа:** Tailscale для зовнішнього доступу
- **USB:** OTG адаптер + libusb passthrough в VM для програматорів STM32/ESP32

### Android — без KVM (стандартний Android, без модифікацій)
Реалістичний варіант для більшості телефонів в апокаліпсисі. Без апаратної віртуалізації — використовується proot (chroot-подібне середовище без root).

- **Хост:** Termux + Termux:Boot + Termux:Widget
- **"VM":** proot-distro + Debian ARM64 (user-space емуляція, не справжня VM)
  ```
  pkg install proot-distro
  proot-distro install debian
  proot-distro login debian
  ```
- **Веб-розробка:** Node.js, npm, pm2, nginx, git (через apt в Debian)
  - Docker — **не працює** в proot (потребує kernel features); альтернатива — podman (з обмеженнями)
- **Системна розробка:**
  - `apt install build-essential clang cmake ninja-build`
  - STM32: `apt install gcc-arm-none-eabi openocd`
  - ESP32: ESP-IDF toolchain, esptool.py
  - QEMU — тільки user-mode (`qemu-user-static`), не system-mode
  - Buildroot/Yocto — працює, але повільно
- **IDE:** code-server або Neovim у терміналі
- **USB:** `pkg install libusb` в Termux + `termux-usb` для passthrough; без root — обмежено, з root (Magisk) — стабільніше
- **Обмеження:** повільніше за KVM, немає Docker, обмежений доступ до /dev без root

### iPhone — з джейлбрейком (A12+ + Hypervisor.framework)
Повний стек, рівнозначний Android з KVM.

- **Хост:** UTM
- **VM:** UTM VM → Debian ARM64 (апаратна віртуалізація через Hypervisor.framework)
- **Веб-розробка:** Node.js, npm, pm2, nginx, Docker, git
- **Системна розробка:**
  - gcc, g++, clang, make, cmake, ninja
  - Linux kernel source + build tools
  - GDB, Valgrind, strace, ltrace
  - Wireshark/tshark, tcpdump
  - STM32: arm-none-eabi-gcc, OpenOCD
  - ESP32: ESP-IDF toolchain, esptool.py
  - QEMU для емуляції без реального заліза
  - Buildroot або Yocto
- **IDE:** code-server або будь-який десктопний через X11/VNC або Neovim
- **Мережа:** Tailscale
- **USB:** обмежено через iOS — потребує додаткового дослідження ⚠️

### Порівняльна таблиця

| Можливість | Android KVM | Android proot | iPhone JB |
|-----------|------------|--------------|-----------|
| Повноцінна VM | ✅ | ❌ (proot) | ✅ |
| Docker | ✅ | ❌ | ✅ |
| USB / OTG | ✅ | ✅ (з root) | ⚠️ обмежено |
| STM32 / ESP32 | ✅ | ✅ | ✅ (без USB) |
| Потребує модифікацій | unlocked BL | ні (root опц.) | джейлбрейк |
| Root потрібен | опціонально | опціонально | — |

## ФАЗА 0 — FUEL (ЖИВЛЕННЯ)

Задача: забезпечити стабільне живлення для телефону в умовах апокаліпсису.
Мова не про акумулятори (вони є одразу), а про **генерацію** енергії.

### Варіант A — Сонячна панель
Мінімум для одного телефону: панель 10-20W + USB контролер заряду.
Телефон споживає ~2-5W при активному використанні, ~0.5W в режимі очікування.

Регіональна специфіка (пікових сонячних годин на день):

| Регіон | Літо | Зима | Примітка |
|--------|------|------|----------|
| Україна / Центральна Європа | 6-8 год | 1-2 год | Взимку критично мало |
| Північна Америка (середня смуга) | 5-7 год | 2-3 год | |
| Північна Америка (південь, Техас/Каліфорнія) | 7-9 год | 4-5 год | |
| Австралія | 6-8 год | 4-6 год | Зима м'яка |
| Індія (середня) | 6-8 год | 5-7 год | Сезон мусонів — проблема |
| Китай (північ) | 5-7 год | 3-4 год | |
| Китай (південь) | 5-7 год | 4-6 год | |
| Японія / Корея | 4-6 год | 2-4 год | |
| Скандинавія | 6-8 год | 0-1 год | Взимку майже нічого |

**Висновок:** в більшості регіонів взимку сонця недостатньо — потрібен backup.

### Варіант B — Велогенератор (саморобний)
Найпростіший саморобний генератор. Не потребує нічого крім велосипеда і базових деталей.

Компоненти:
- Велосипедне динамо (або безколекторний мотор від дрона/принтера як генератор)
- Діодний міст (4 діоди 1N4007) — випрямляч змінного струму
- Конденсатор 1000мкФ 25V — згладжування
- Стабілізатор LM7805 або DC-DC buck converter — вихід 5V
- USB роз'єм

Людина педалює 1 годину → ~50-100Wh → заряджає телефон повністю + запас.

### Варіант C — Вітрогенератор
Реалістичний якщо є постійний вітер (узбережжя, відкрита рівнина, гори).
Мінімальна конструкція: ротор Савоніуса (вертикальна вісь) — простий у виготовленні, не потребує орієнтації на вітер.
При вітрі 5 м/с ротор 1м² дає ~10-20W.

### Варіант D — Водяний млин / мікроГЕС
Якщо є річка або струмок з перепадом висоти.
Навіть маленький потік 10л/с з висотою 1м дає ~50-100W.
Перевага: працює 24/7 незалежно від погоди.
Конструкція: колесо або турбіна → генератор → випрямляч → стабілізатор.

### Варіант E — Автомобільний генератор
Якщо є автомобіль з паливом — генератор вже є вбудований.
12V від акумулятора → DC-DC converter → 5V USB.
Проблема: паливо скінчиться, але на перехідний період це найпростіший варіант.

## ФАЗА 1 — SPARK (ПЕРШИЙ LINUX)

**Android:** Termux з DEPOT (офлайн APK) → gcc, python, git → повноцінне Linux середовище.
**iPhone:** UTM (з джейлбрейком) або обмежений варіант через iSH/a-Shell (без джейлбрейку).

Вхід: телефон з живленням, є COLDSTART-DEPOT на флешці або SD карті.
Вихід: працює gcc, python, git, DEPOT доступний локально.

## ФАЗА 2 — HORN (ЛОКАЛЬНА МЕРЕЖА, клан 2-200 людей)

Вхід: є хоча б 2 смартфони (Android або iPhone).
Вихід: захищена локальна мережа для клану, текстова + голосова комунікація.

**Обмеження:** ніяких операторів, ніяких базових станцій. Тільки пристрій-до-пристрою.
**Масштаб:** до ~200 людей (Dunbar's number 150 + запас).

### Технологія: Jami
- Децентралізований P2P месенджер, без центрального сервера
- Працює в локальній мережі без інтернету
- Підтримує текст + голосові дзвінки + відео
- **Доступний на Android І iOS** (без джейлбрейку)
- Open source

### Технологія: Briar (тільки Android)
- Mesh через WiFi Direct + Bluetooth
- Працює без інтернету і без root
- Групові чати + приватні чати (текст)
- Радіус: Bluetooth ~30м, WiFi Direct ~100м
- Перевага перед Jami: справжній mesh, не потребує точки доступу

### Стратегія: гібрид
- **Briar** — для Android-to-Android mesh без точки доступу
- **Jami** — для змішаних груп Android + iPhone через локальну WiFi мережу (один телефон як hotspot)
- iPhone без джейлбрейку не підтримує WiFi Direct і Bluetooth mesh — тільки через hotspot

**Важливо:** 200 людей в радіусі 100м — це компактний табір або будівля.
Для більшої площі потрібні стаціонарні телефони-ретранслятори (на будівлях, деревах) з живленням від Фази 0.

## ФАЗА 3 — REACH (РЕГІОНАЛЬНА МЕРЕЖА)

Вхід: кілька локальних мереж кланів (Фаза 2).
Вихід: об'єднана регіональна мережа, комунікація між кланами на відстані кілометрів.

**Фундаментальна проблема:** WiFi 2.4GHz — максимум 300-500м у відкритому полі.
Покрити регіон тільки мобільними телефонами фізично неможливо без ретрансляторів.

### Рішення: гібридна mesh

**Рухомі вузли** — люди з телефонами. Синхронізуються при зустрічі (store-and-forward).
**Стаціонарні ретранслятори** — телефони або дешеві Android пристрої, встановлені на висоті (дахи, щогли) з живленням від сонячних панелей (Фаза 0). Обов'язкова технологія для регіонального покриття.

### Store-and-forward (принцип пошти)
Повідомлення подорожує від людини до людини при фізичній зустрічі або при входженні в зону покриття.
Повільно (години/дні для великих відстаней) але надійно.
Briar підтримує цю модель нативно.

### Об'єднання кланів
Фізична зустріч представників → обмін ключами/контактами → federation локальних mesh мереж.
Вхідна точка Фази 3: спочатку фізичний контакт, потім цифрове об'єднання.

## ФАЗА 4 — SPINE (ІНФРАСТРУКТУРА)

Вхід: регіональна мережа існує. З'являються інші пристрої (RPi, ноутбуки, старе залізо).
Вихід: стаціонарна інфраструктура, що підсилює мережу і надає базові сервіси.

Нові пристрої входять в існуючу мережу як вузли.

**Мінімальний набір:**
- Ретранслятор (підсилення mesh покриття)
- Офлайн довідник (COLDSTART, доступний всім по мережі)
- Офлайн карти (OpenStreetMap)
- База даних (хто живий, де що є, медичні записи)
- Сервер збереження повідомлень

---

## ФАЗА 5 — GUILD (СЕРВІСИ)

Вхід: стабільна інфраструктура Фази 4.
Вихід: цифрові сервіси для суспільства.

**Пріоритети (від критичного до бажаного):**
1. **Базові реєстри** — хто живий, медичні записи, облік ресурсів
2. **Обмін ресурсами** — бартерний облік, ринок
3. **Банкінг** — потребує ідентифікації і криптографії
4. **Державні/адміністративні сервіси** — реєстрація, законодавство
5. **Галузеві сервіси** — медицина, освіта, логістика
6. **Ентертейнмент** — радіо, відео, ігри
