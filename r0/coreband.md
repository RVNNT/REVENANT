SX1280 (LoRa 2.4GHz) — найцікавіший варіант. Той самий форм-фактор, RadioLib підтримує, але в 40 разів більша пропускна здатність і апаратний FHSS. Мінус — менша дальність і конкурує з WiFi/BLE за 2.4GHz.

802.11s WiFi mesh між стаціонарними вузлами (будівлі, ліхтарні стовпи). LoRa тільки як резервний канал і для вузлів без електрики.
Проекти що роблять це: Althea, RightMesh, Meshtastic + WiFi hybrid.

Як це зробити реально? (Стратегія Фази 5)
Щоб замінити 2G/3G без вишок операторів, твоя система має стати "Клітинним Мешем" (Cellular Mesh).

А. Просторова ізоляція (Spatial Reuse)
Ми маємо використовувати те, що LoRa на SF7 у місті б'є лише на 1-1.5 км.
Люди в одному районі не чують людей в іншому.
Ідея: Частоти мають використовуватися повторно (як у сотах). Клан у районі А і Клан у районі Б працюють на одній частоті, але не заважають один одному через відстань.

Б. Ієрархія: "Клани" та "Магістралі"
Замість того, щоб кожен телефон був ретранслятором для всіх, ми ділимо ролі:
Вузли (Користувачі): Працюють на мінімальній потужності, чують тільки свій "Клан" (сусідів у радіусі 300-500м).
Супер-вузли (Community Infrastructure): Ентузіасти ставлять донгли з потужними антенами на дахах. Вони стають "містками" між районами.
Маршрутизація (Distance Vector Routing): Пакет не летить усім підряд. Він знає шлях: Вузол -> Супер-вузол району -> Супер-вузол центру -> Вузол отримувача.

## 0. ГОЛОВНИЙ ПРИНЦИП

**Фаза 1 — COREBAND є стандартним RNode.** Sideband, Reticulum, будь-який RNS-сумісний клієнт підключається і працює без змін.

**Фаза 2+ — COREBAND розширює протокол.** CLAN-розширення додаються поверх стандартного RNode протоколу. Sideband ігнорує невідомі команди і продовжує працювати.

**База коду:** COREBAND не пишеться з нуля. Фаза 1 базується на існуючому RNode firmware коді. CLAN шари додаються поверх як окремі модулі. Це гарантує стабільність і швидкість старту.

Це гарантує:
- тестування заліза з першого дня
- незалежність від готовності OUTBAND
- безпечну міграцію без розриву сумісності

---

## 1. ФОРМАЛЬНІ ПРАВИЛА ЕВОЛЮЦІЇ

*Ці три правила є незмінними на всіх фазах. Їх порушення ламає стратегію еволюції.*

**ПРАВИЛО 1: Standard RNode behavior is immutable.**
Стандартні `CMD_*` команди і serial framing не змінюються ніколи. Жодна фаза не модифікує поведінку існуючих RNode команд.

**ПРАВИЛО 2: All CLAN features are feature-gated.**
Кожна CLAN фіча активується тільки після явної ініціалізації (provisioning або команда від OUTBAND). До цього моменту — поведінка стандартного RNode без винятків.

**ПРАВИЛО 3: CORE-mode і RNode-mode співіснують.**
COREBAND завжди може повернутись в RNode-mode якщо security-функції не ініціалізовані або при factory reset. Жодна фаза не забирає цю можливість.

## 2. АПАРАТНА БАЗА

- **MCU:** ESP32 (dual-core, 240 MHz)
- **Радіо:** SX1262 (основний), SX1278 (альтернатива)
- **Діапазон:** 433 MHz (базовий), оперативна частота визначається NOISE-KILLING налаштуваннями
- **Бібліотека радіо:** RadioLib
- **З'єднання з телефоном:** BLE + USB-OTG (serial)
- **Пам'ять:** NVS (Non-Volatile Storage) для конфігурації та ключів
- **Живлення:** Light Sleep / Deep Sleep, ціль < 50mA в очікуванні

## 3. АРХІТЕКТУРНІ ІНВАРІАНТИ — ОБОВ'ЯЗКОВО З ФАЗИ 1

Ці абстракції закладаються одразу і не переписуються:

```
Application / CLAN Logic
        ↓
   CLAN MAC Layer
        ↓
 Radio Abstraction Layer
   get_active_channel(timestamp)
        ↓
   SX1262 Driver (RadioLib)
```

**Правила:**
- `setFrequency()` викликається **тільки** з Radio Abstraction Layer
- MAC не знає що частота одна — тільки `get_active_channel(timestamp)`
- Фаза 1: `get_active_channel()` завжди повертає `channel_0`
- Часова модель існує з фази 1 (frame tick, guard time, TX window)
- **У фазі 1 MAC є прозорим** — він не перехоплює і не модифікує стандартні RNode пакети. Весь трафік проходить через MAC без змін. Інакше фаза 1 вже не є стандартним RNode.

## 4. ФАЗА 1 — СТАНДАРТНИЙ RNODE + NOISE-KILLING БАЗОВИЙ

*Результат: COREBAND повністю сумісний з Sideband та будь-яким RNS клієнтом. Мінімальний interference з публічними мережами.*
*Орієнтовний термін: 4–6 тижнів.*

### 4.1. База коду

COREBAND фази 1 = існуючий RNode firmware + CLAN abstraction шари поверх. Не форк, не переписування. Додається:
- Radio Abstraction Layer
- Channel Abstraction (`get_active_channel()`)
- Мінімальна часова модель
- CLAN Handshake обробник (окремий control channel, не втручається в радіопотік)

### 4.2. Реалізація стандартного RNode протоколу

COREBAND реалізує **повний протокол RNode** від Mark Qvist:

- Serial framing (HDLC-подібний)
- Команди: `CMD_FREQUENCY`, `CMD_BANDWIDTH`, `CMD_SF`, `CMD_CR`, `CMD_TXPOWER`, `CMD_RADIO_STATE`
- Отримання та передача LoRa пакетів
- Відповідь на `CMD_DETECT` — ідентифікація як RNode
- RSSI та SNR в заголовку кожного прийнятого пакета

**Перевірка фази 1:** Sideband підключається через USB-OTG або BLE Serial і бачить стандартний RNode. Текстовий зв'язок через Reticulum працює.

### 4.3. Транспорти підключення

**USB-OTG (Serial):**
- CDC ACM або UART через USB
- Стандартний для RNode — працює з Sideband без налаштувань
- Baudrate: 115200

**BLE Serial:**
- Nordic UART Service (NUS) — стандартний BLE serial профіль
- Sideband підтримує BLE RNode через NUS
- Оптимізація MTU для стабільної передачі (підготовка до голосу у фазі 2)

### 4.4. Радіо конфігурація + NOISE-KILLING Фаза 1

**Базові параметри:**
- Single frequency (channel_0) через Radio Abstraction Layer
- LBT: CAD (Channel Activity Detection) замість простого RSSI
- Random Backoff: 100–500 мс якщо канал зайнятий
- Конфігурується через стандартні RNode команди

**NOISE-KILLING — Оптимізація передачі:**

*SF7 / BW 250 kHz (замість SF9-SF11):*
Пакет стає в 4 рази швидшим — шанс колізії падає пропорційно. Чутливості SF7 у місті достатньо для 1–1.5 км. Додатковий ефект: SF7 і Meshtastic SF11 мають різницю у 4 ступені — SX1262 відкидає чужий сигнал як шум (~15 dB rejection), фізично не бачить Meshtastic трафік як колізію.

*Frequency Offset — channel_0 = 434.000 MHz і вище:*
Meshtastic за замовчуванням: 433.175 MHz. Зсув менше 200 кГц недостатній для смуги BW250. Перехід на 434.000 MHz і вище дозволяє повністю вийти з «дзвону» Meshtastic-каналів.

*CAD — Channel Activity Detection:*
Замість вимірювання шуму (RSSI) чіп шукає LoRa-преамбулу апаратно (`radio.scanChannel()` в RadioLib). Сигналізації та FSK-сміття більше не тригерять backoff. CLAN працює у паузах між LoRa трафіком, ігноруючи весь не-LoRa шум на частоті.

**Ефект NOISE-KILLING Фаза 1:** поріг витримуваних сусідніх Meshtastic-вузлів зростає з ~210 до ~660 у радіусі 1.5 км.

### 4.5. Часова модель (мінімальна)

Закладається з фази 1:
- Внутрішній frame tick
- Guard time між передачами
- TX window abstraction

Функціонально непомітна у фазі 1, критична для фази 4.

### 4.6. CLAN Handshake ідентифікатор

CLAN Handshake — окремий control channel поверх serial. Він не втручається в радіопотік, не змінює state машину RNode, не впливає на радіоповедінку. Стандартний Sideband цей запит не надсилає і повністю його ігнорує.

При отриманні CLAN-специфічного запиту від OUTBAND — COREBAND відповідає:

```json
{
  "hw": "COREBAND",
  "fw_version": "1.0.0",
  "clan_phase": 1,
  "capabilities": ["text"]
}
```

### 4.7. NVS структура (закладається з фази 1)

Навіть якщо більшість полів порожні — структура NVS визначається одразу:

```
nvs/
  clan/
    hw_id          ← унікальний ID пристрою
    fw_version     ← версія прошивки
    clan_phase     ← поточна фаза (1)
    radio/
      channel_0    ← базова частота (434.000+ MHz)
      bandwidth    ← 250 kHz
      sf           ← SF7
      cr
      tx_power
    security/      ← порожньо у фазі 1, заповнюється у фазі 3
      ifac_key
      group_key
      epoch
    blacklist/     ← порожньо у фазі 1
```

### 4.8. Що НЕ робиться у фазі 1

- Немає Panic PIN на рівні прошивки (тільки в OUTBAND)
- Немає NVS wipe по команді
- Немає шифрування на рівні прошивки (все через Reticulum/OUTBAND)
- Немає FHSS
- Немає голосу

## 5. ФАЗА 2 — ГОЛОС, QoS ТА NOISE-KILLING РОЗПОДІЛ

*Результат: PTT, голосові повідомлення, оптимізований BLE для аудіо, розподіл каналів голос/текст.*
*Орієнтовний термін: +4–6 тижнів після фази 1.*

### 5.1. BLE MTU оптимізація

- Збільшення MTU до максимально підтримуваного (517 байт для BLE 4.2+)
- Потоковий режим для Codec2 аудіо
- Буферизація аудіо пакетів на стороні COREBAND

### 5.2. Голосовий pipeline

**PTT (Stream):**
- SF9 / 250kHz
- Прийом аудіо потоку від OUTBAND через BLE
- TOT контроль: 10 сек максимум, countdown за 3 сек до відсічення
- Guard Time: 3 сек після відпускання PTT

**Voice Messages:**
- SF11 для надійності
- Фрагментація і збірка пакетів на рівні прошивки
- Максимум 10 сек запису

____===== **OR** ======_____
**PTT (Stream) + Voice Messages:**
- custom lightweight UDP-like datagram over raw RF


### 5.3. QoS на рівні прошивки

**Важливо:** QoS і пріоритизація застосовуються **тільки до CLAN-маркованого трафіку**. Стандартні RNode пакети проходять через прошивку без змін — як у фазі 1. Це гарантує коректну поведінку COREBAND у змішаній мережі.

Черга передачі CLAN трафіку з пріоритетами:
```
1. Voice Stream (PTT)     ← абсолютний пріоритет
2. Voice Message          ← високий
3. Text (LXMF)            ← нормальний
4. Beacon                 ← низький
```

При активному PTT — прошивка призупиняє обробку CLAN черг 3 і 4.

### 5.4. Airtime Budget

- Rate limit per identity (відстежується за source hash пакета)
- Beacon suppression при congestion (RSSI + queue depth)
- Flood protection

### 5.5. NOISE-KILLING Фаза 2 — Дисципліна та розподіл

**Dual Channel — розділення голосу і тексту:**
Текстовий трафік і маяки йдуть на `channel_0` (434.000+ MHz). При PTT донгл автоматично перемикається на `channel_voice` (наприклад, 434.200 MHz). Після завершення PTT — повернення на `channel_0`. Перемикання реалізовується через Radio Abstraction Layer — `setFrequency()` тільки звідти. Голос не душить текстові повідомлення, текст не перериває рацію.

**Smart Airtime Guard:**
Жорсткий програмний ліміт PTT: не більше 10 секунд, після чого передавач блокується на 3 секунди (Guard Time). Ніхто не завісить канал випадковим натисканням кнопки в кишені.

**Fragmentation без надлишкового FEC:**
Голосові повідомлення ріжуться на шматки по 20–40 мс. LoRa має вбудований CR (Coding Rate) — додатковий software FEC збільшує ToA без суттєвого приросту надійності в міських умовах. Якщо один пакет втрачається через колізію — слухач почує мікро-тріск, а не втратить все речення.

**Ефект NOISE-KILLING Фаза 2:** Dual Channel фактично подвоює пропускну здатність для сценарію голос+текст. Поріг витримуваних Meshtastic-вузлів зростає до ~850.

## 6. ФАЗА 3 — БЕЗПЕКА НА РІВНІ ЗАЛІЗА + NOISE-KILLING АВТОМАТИЗАЦІЯ

*Результат: NVS wipe, IFAC, Blacklist на залізі, Smart Beacon, BLE Gateway, автономне управління спектром.*
*Орієнтовний термін: +4–6 тижнів після фази 2.*

Всі функції цієї фази активуються **тільки після provisioning** (ПРАВИЛО 2). До ініціалізації COREBAND поводиться як стандартний RNode.

### 6.1. NVS Wipe (Hardware Panic)

При отриманні підписаного wipe-пакета від OUTBAND (Panic PIN або Remote Kill-switch):

```
1. Zeroization RAM (перезапис нулями)
2. NVS erase (повне затирання)
3. BLE disconnect
4. Reboot в factory state (= RNode-mode, ПРАВИЛО 3)
```

Вся операція — не більше 3 секунд.

### 6.2. IFAC — апаратний фільтр

- 4-байтовий фільтр на рівні SX1262
- Ключ отримується від OUTBAND при provisioning, зберігається в NVS
- Пакети без валідного IFAC відкидаються на рівні заліза до передачі в Reticulum
- Активується тільки після provisioning. До цього — фільтрація відсутня.
- Оновлення при Group Rekeying — підписаний пакет від ADMIN

### 6.3. Blacklist на рівні прошивки

Апгрейд від software Blacklist в OUTBAND:

- Список заблокованих Identity hash зберігається в NVS
- Перевірка відбувається до передачі пакета в BLE/USB буфер
- Оновлення через підписаний revoke packet від OUTBAND
- Максимум 64 записи у фазі 3

### 6.4. Smart Beacon

Автономна робота без участі телефону:

- Базова частота: 15 хв
- Jitter: ±60 сек (генерується апаратним RNG ESP32)
- Вміст: hw_id, role hash, battery level, RSSI
- Автоматичне пригнічення при congestion

### 6.5. BLE Gateway — датчики

- Автономне опитування периферії через BLE (датчики руху, розтяжки)
- Незалежно від стану підключення телефону
- При спрацюванні: генерація тривожного пакета та передача в LoRa мережу
- Service Diagnostics: Noise Floor, SNR — доступно через OUTBAND

### 6.6. READ MODE підтримка

При отриманні команди READ MODE від OUTBAND:

- TX повністю вимикається (апаратно через RadioLib)
- RX залишається активним
- Heartbeat виняток: раз на 6 годин мінімальний підписаний пакет присутності

### 6.7. Синхронізація RTC

- При кожному BLE підключенні OUTBAND передає timestamp
- COREBAND оновлює внутрішній RTC
- Точність RTC критична для фази 4

### 6.8. NOISE-KILLING Фаза 3 — Автоматична втеча

**Background Noise Scan:**
У паузах між роботою донгл сканує сусідні частоти ISM-смуги і вимірює Noise Floor через RSSI. Результати зберігаються як карта завантаженості каналів і доступні через OUTBAND для аналізу та відображення.

**Auto-Frequency Agility — легкий FHSS:**
Якщо Noise Floor поточного каналу перевищує поріг 50% завантаженості протягом заданого вікна часу — COREBAND генерує сповіщення для OUTBAND. ADMIN отримує пуш: *«Ефір брудний. Переходимо на Канал 3?»* Після підтвердження ADMIN-ом COREBAND розсилає міграційний beacon з новою частотою — весь Клан синхронно перемикається за один цикл. Перемикання відбувається через `get_active_channel()` — Radio Abstraction Layer залишається незмінним.

**Ефект NOISE-KILLING Фаза 3:** система автономно реагує на деградацію спектру без участі зв'язківця. Є підготовка інфраструктури (time sync, channel switching) для повного FHSS фази 4.

## 7. ФАЗА 4 — FHSS (ПОВНИЙ NOISE-KILLING)

*Результат: frequency hopping, повний захист від РЕБ/РЕР, невразливість до колізій із тисячами пристроїв.*
*Орієнтовний термін: +6–8 тижнів після фази 3.*

> **Якщо фази 1–3 реалізовані правильно** — фаза 4 це зміна однієї функції і розширення планувальника. Жодна бізнес-логіка не переписується.

### 7.1. Два режими роботи (ПРАВИЛО 3)

COREBAND фази 4 має два явних режими:

**CORE-mode:**
- FHSS активний
- Hop table на основі групового ключа
- Повний захист від РЕБ/РЕР
- Активується після provisioning з FHSS параметрами

**RNode-mode:**
- Фіксована частота (channel_0)
- Повна сумісність зі стандартними RNode клієнтами
- Активується при відсутності provisioning або після factory reset

Перемикання між режимами — через команду від OUTBAND або автоматично при factory reset.

### 7.2. Slot Model (TDMA)

- Fixed slot duration ~50 мс
- Global time model на базі синхронізованого RTC (закладено у фазі 3)
- OTA time sync між вузлами через спеціальний sync пакет
- Drift compensation (накопичена похибка RTC)
- Кожен вузол точно знає, коли йому дозволено «стрибнути»

### 7.3. Hop Table

- FHSS seed = функція від групового ключа (NVS) + epoch
- Детермінована hop sequence: однаковий seed → однакова послідовність на всіх вузлах
- Сигнал «розмазується» по діапазону 433.000–434.500 MHz (~13 каналів по 125 kHz)
- Guard window між стрибками
- Channel settle time для SX1262 (~1 мс)

Приклад hop sequence:
```
Слот 1 → 433.100 MHz
Слот 2 → 434.200 MHz
Слот 3 → 433.800 MHz
...
```

### 7.4. Radio Abstraction Layer — повна версія

```cpp
// Фаза 1–3 (RNode-mode або channel_0):
float get_active_channel(uint32_t timestamp) {
    return channel_0;
}

// Фаза 4 (CORE-mode, FHSS):
float get_active_channel(uint32_t timestamp) {
    uint32_t slot = timestamp / SLOT_DURATION_MS;
    return hop_table[slot % HOP_TABLE_SIZE];
}
```

### 7.5. MAC Scheduler — повна версія

```cpp
for each slot:
    tune_radio(get_active_channel(now()))
    if LBT_clear():
        if tx_window_open():
            transmit_next_from_queue()
```

Уся логіка вище (QoS, PTT, Beacon, Dual Channel) залишається незмінною.

### 7.6. Ефект NOISE-KILLING Фаза 4

```
Meshtastic сидить на одній частоті.
CLAN стрибає по 13 каналах.
На кожному окремому стрибку CLAN конкурує з Meshtastic
лише 1/13 часу свого циклу.

Поріг витримуваних Meshtastic-вузлів: ~11,700 у радіусі 1.5 км.
Проблема interference перестає існувати як інженерна задача.
```

## 8. FIRMWARE MANAGEMENT

### 8.1. Перша прошивка (factory flash)

- Через USB-OTG за допомогою `esptool.py`
- OUTBAND надає інструкцію та файл прошивки з CLAN DEPOT
- Не потребує окремого програматора

### 8.2. OTA оновлення через BLE

- OUTBAND передає бінарник прошивки через BLE chunks
- COREBAND верифікує підпис перед застосуванням
- Rollback до попередньої версії якщо boot після оновлення не відбувся

### 8.3. Snapshot / Rollback

- NVS Snapshot: збереження поточної конфігурації перед оновленням
- Rollback N-1: повернення до попередньої прошивки + конфігурації
- Legacy Backup через USB-OTG: повний дамп NVS на телефон
